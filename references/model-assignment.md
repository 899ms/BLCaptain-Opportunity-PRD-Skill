# 动态模型分工规则

## 核心原则

模型分工由用户实际配置的外部模型决定。Codex 是主持人和最终整合者，不计入外部模型能力池。

## 角色池

| 角色 | 任务 | 不承担 |
|---|---|---|
| 主分析 | 归纳评论、抽取痛点、生成初步机会假设 | 不单独决定 Go |
| 长文本理解 | 压缩长评论、拆意图、保留用户原话 | 不编造证据 |
| 商业反方 | 评估 ROI、CAC/LTV、付费意愿、替代成本 | 不替代真实付费证据 |
| 结构化审查 | 组织方法论、整理 PRD 结构、检查中文语境 | 不决定商业价值 |
| 外部视角 | 趋势语境、社交讨论、反向证据补充 | 不作为唯一证据 |
| 代码实现 | 技术边界、7 天 demo 可行性、开发交接 | 不决定是否值得做 |

## 分配规则

| 可用外部模型数 | 分工 |
|---:|---|
| 0 | 输出配置引导 |
| 1 | 单模型低置信度模式：一个模型做分析，主 Agent 做主持和校验 |
| 2 | 主分析 + 反方审查 |
| 3 | 主分析 + 商业反方 + 结构化审查 |
| 4+ | 多视角讨论：主分析、长文本、商业反方、结构化、外部视角、代码实现按能力标签分配 |

## 选择算法

1. 优先使用用户手动标记的用途标签。
2. 没有标签时，根据模型名称做保守推断，并标注“系统推断”。
3. 多个模型适合相同角色时，优先分配给健康检查结果更稳定的模型。
4. 角色不足时允许合并，但必须在报告中写明缺口。
5. 主 Agent 始终负责最终综合、文件生成、校验和交接。
6. `codex_builtin` 只表示主持可用，不参与外部模型角色分配。

## 输出格式

```markdown
## 动态模型分工

| 模型 | 调用方式 | 健康状态 | 能力标签 | 本轮角色 | 分配依据 |
|---|---|---|---|---|---|
```

## 禁止项

- 不写“某个固定模型必须负责长文本”这类固定规则。
- 不写“某个固定模型必须否决”这类固定规则。
- 不把模型观点当事实证据。
