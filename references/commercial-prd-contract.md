# 商业化机会 + 工程实施 PRD 合同

## 触发条件

只有机会评估报告判断为 Go，且 G8 通过时，才生成最终 PRD。

如果原始命题是 Watch 或 Pivot，但证据已经集中到一个明确痛点簇，必须先执行 Cut-to-Go：把宽泛命题收窄为一个最小商业切口，重新跑 Gate。只有收窄切口 Go 且 G8 通过时，才生成最终 PRD。

最终 PRD 不是机会说明文档，也不是任何外部 PRD 生成器的下游产物。它必须把“为什么值得做”和“怎么工程实施”写在同一份可交接文档里，供产品、设计、研发、测试和后续实现直接使用。

## 章节

1. 商业速读卡
2. 机会摘要
3. 证据墙
4. 目标用户与核心 JTBD
5. 当前替代方案与竞品缺口
6. 需求分析与方法论选择
7. MVP 功能边界
8. 商业模型与增长路径
9. 产品交互与核心流程
10. 工程实施方案
11. 数据模型与字段字典
12. API 契约
13. 权限、安全、隐私与合规
14. 非功能需求
15. 异常流程和边界条件
16. 测试方案和验收标准
17. 部署运维与监控
18. 埋点与验证指标
19. 开发任务拆分与 DoD
20. 7 天验证计划
21. 风险、反证与停止条件
22. AI 开发者交接说明

## 硬约束

- P0 只保留 1 到 3 个用户故事。
- 每个 P0 用户故事必须包含 evidence_id。
- 每个 Go 结论必须有反向证据回应。
- 必须包含 7 天关键假设实验。
- 必须包含至少 3 条验收剧本。
- 必须写清停止条件。
- 必须包含系统架构、数据流、技术选型和模块边界。
- 必须包含数据字段字典，至少覆盖核心实体、字段名、类型、来源、保留策略。
- 必须包含 API 契约，至少覆盖路径、方法、请求字段、响应字段、错误码。
- 必须包含 AI 或规则分析方案，写清标签、置信度、低置信度处理、人工复核和成本上限。
- 必须包含权限、安全、隐私、删除策略和审计日志。
- 必须包含非功能需求，至少覆盖性能、容量、可用性、成本、数据保留。
- 必须包含异常流程，至少覆盖导入失败、空数据、低置信度、分析超时、导出失败、数据过期。
- 必须包含测试方案，至少覆盖单元测试、集成测试、端到端验收、人工评估样本。
- 必须包含部署运维、监控告警、日志和回滚策略。
- 必须包含开发任务拆分和 DoD，每个任务能被研发独立领取。

## 机会评估报告优先

未 Go 时输出机会评估报告，不输出商业化机会 PRD。机会评估报告必须包含：

- 决策：Go、Watch、Pivot 或 No-Go
- 决策理由
- 不生成商业化 PRD 的具体 Gate 原因
- 正向证据摘要
- 反向证据摘要
- 痛点簇和候选切口，如果证据已经集中
- Gate 结果
- 下一步动作
- 停止或补证条件

## AI 开发者交接

交接说明必须告诉实现者：

- 先做哪个 P0
- 哪些内容不能重解释
- 哪些地方可以自由优化
- 哪些事实仍然未知
- 如何验证第一版
- 推荐的第一批文件、模块或接口
- 不允许在没有新证据时扩展的范围
- 生成代码前必须先跑哪些校验

## 工程可实施判定

一份 Go 后最终 PRD 只有同时满足以下条件，才算可指导后续开发：

- 研发能根据 API 契约和字段字典搭建第一版服务。
- 测试能根据验收剧本和异常流程写测试用例。
- 产品能根据 MVP 边界判断哪些需求不能加入 P0。
- 运维或工程负责人能看到部署、监控、成本和回滚策略。
- Codex 能根据开发任务拆分继续实现，不需要重新发明技术方案。
