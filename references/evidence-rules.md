# 证据规则

## 有效证据

一条有效证据至少包含：

- evidence_id，例如 E-001
- 平台
- 日期
- URL 或可追溯来源
- 用户原话
- 行为信号
- 痛点标签

商业化判断还需要至少一种商业信号：

- 已付费或明确报价
- 弃用竞品
- 付费抱怨
- 成本节省
- 购买意图
- 替代人工或外包

## 证据分级

| 级别 | 条件 | 用途 |
|---|---|---|
| A | 原话、URL、日期、手动行为、商业信号齐全 | 可支持 Gate 和 P0 |
| B | 原话、URL、日期、行为齐全，商业信号未知 | 可支持痛点，不支持商业化 Go |
| C | 有抱怨但无行为 | 只作线索 |
| 背景 | 超出 30 天窗口或缺少当前性 | 只作背景 |
| 排除 | 广告、软文、公告、转述、重复账号、水军迹象 | 不参与评分 |

## 反向证据

Go 前必须主动寻找反向证据：

- 免费替代是否足够好
- 竞品是否已经覆盖
- 用户是否没有购买权
- 合规或隐私是否阻断
- 痛点是否只是一次性抱怨
- 获客渠道是否不可达

反向证据使用 reverse_id，例如 R-001，并逐条回应。

## 证据墙字段

```markdown
| evidence_id | 平台 | 日期 | URL | 用户原话 | 行为信号 | 商业信号 | 标签 | 等级 |
|---|---|---|---|---|---|---|---|---|
```

## 输出纪律

- 保留用户原话，不改写成产品经理语言。
- 推断必须标注为推断。
- 未知必须写“未知”，不能补全。
- P0 功能必须能追溯到 evidence_id。
